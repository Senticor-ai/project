# Epic: Docker Image Security CVEs

**Status:** In progress — three remediations applied 2026-03-03
**Last reviewed:** 2026-03-03
**Source:** [GitHub Code Scanning](https://github.com/Senticor-ai/project/security/code-scanning) (Trivy container scan)

---

## Summary

Our nightly security scan (Trivy) flags **~144 CVEs** across the three Docker images
we build. Most are in OS-level packages inherited from base images (`python:3.12-slim`,
`nginx:1.27-alpine`). None have upstream fixes available at time of writing.

This epic documents every CVE, assesses actual risk in our deployment context, and
lists remediation approaches for the team to evaluate.

## Threat Assessment

### What is actually exposed?

Our containers run in a **Kubernetes cluster** with the following security controls already in place:

1. **Non-root execution** — All containers run as non-root users (`USER appuser` / `USER 101`)
2. **Dropped capabilities** — `capabilities.drop: ["ALL"]` on all pods
3. **No privilege escalation** — `allowPrivilegeEscalation: false`
4. **Seccomp profiles** — `RuntimeDefault` on all pods
5. **Network policies** — Pods only accept traffic from the ingress controller
6. **No shell access** — End users cannot execute commands inside containers
7. **Read-only concerns** — Frontend serves static files via nginx; no user-uploaded content parsed by libxml2/libpng

### Risk by category

| Package Group                           | Affected Images   | Actual Risk    | Rationale                                                                                                                                                                       |
| --------------------------------------- | ----------------- | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **OpenSSL** (critical/high)             | Frontend          | **Low**        | nginx terminates TLS at the ingress controller, not inside the container. OpenSSL is present but not used for TLS termination. PKCS#12/PKCS#7 features are not used.            |
| **libxml2** (critical/high)             | Frontend, Backend | **Low**        | Frontend never parses XML. Backend uses it transitively via WeasyPrint for PDF generation from trusted, program-generated HTML — not arbitrary user XML.                        |
| **libpng** (high/medium)                | Frontend, Backend | **Low**        | Frontend serves pre-built static assets. Backend uses libpng transitively via WeasyPrint for images embedded in CVs — input is controlled.                                      |
| **glibc** (high/medium)                 | Backend           | **Medium**     | `memalign` heap corruption (CVE-2026-0861) affects any process using glibc. Agents migrated to Alpine (musl). Backend mitigated by non-root, dropped capabilities, and seccomp. |
| **pypdf** (medium)                      | Backend           | **Resolved**   | Upgraded from 6.6.2 to 6.7.5 — all 6 CVEs fixed.                                                                                                                                |
| **cryptography** (high)                 | Backend           | **Resolved**   | Upgraded from 46.0.4 to 46.0.5 — CVE-2026-26007 (subgroup attack) fixed.                                                                                                        |
| **curl** (medium)                       | Frontend          | **Low**        | curl is in the Alpine image but nginx does not use it at runtime.                                                                                                               |
| **libtiff/cairo/pango** (low)           | Backend           | **Low**        | Used by WeasyPrint for PDF rendering. Input is program-generated HTML, not arbitrary user files.                                                                                |
| **glib/harfbuzz/freetype** (medium/low) | Backend           | **Low**        | Text rendering libraries used by WeasyPrint. Same controlled-input rationale.                                                                                                   |

## Remediation Approaches

### Approach 1: Upgrade base images (recommended first step)

No action needed — CI runs on ephemeral `ubuntu-latest` runners with no Docker
layer cache configured, so every build already pulls fresh base image layers.

**Effort:** None
**Impact:** Automatic on every deploy
**Risk:** N/A

### Approach 2: Switch frontend to distroless nginx

Replace `nginx:1.27-alpine` with a distroless or minimal nginx image that strips
unnecessary packages (libxml2, libpng, curl, openssl CLI tools):

```dockerfile
# Option A: nginxinc/nginx-unprivileged (fewer packages)
FROM nginxinc/nginx-unprivileged:1.27-alpine

# Option B: Build custom minimal image
FROM nginx:1.27-alpine AS base
FROM alpine:3.21
COPY --from=base /usr/sbin/nginx /usr/sbin/nginx
COPY --from=base /etc/nginx /etc/nginx
# Only copy the exact shared libraries nginx needs
```

**Effort:** Medium
**Impact:** Would eliminate ~30+ frontend CVEs (libxml2, libpng, curl, etc.)
**Risk:** Medium — needs thorough testing of nginx functionality

### Approach 3: Switch Python images to Alpine or distroless

**Agents: Done** — migrated to `python:3.12-alpine` on 2026-03-03. The agents image
has zero system package dependencies, making Alpine a safe choice.

**Backend: Blocked** — WeasyPrint depends on cairo/pango/gobject which require
specific system libraries. Alpine builds of these are notoriously fragile.
Evaluate distroless as an alternative (see Approach 6 below).

**Effort:** Done (agents) / High (backend)
**Impact:** Eliminated ~15-20 CVEs from agents image
**Risk:** N/A (agents) / High (backend)

### Approach 4: Multi-stage builds stripping unused packages

Add a final stage that removes packages not needed at runtime:

```dockerfile
FROM python:3.12-slim AS base
# ... install dependencies ...

FROM base AS production
# Remove build-only and unnecessary packages
RUN apt-get purge -y --auto-remove \
    perl perl-base \
    tar \
    coreutils \
    && rm -rf /var/lib/apt/lists/*
```

**Effort:** Medium
**Impact:** Would eliminate ~10-15 CVEs in unused packages
**Risk:** Medium — must verify no runtime dependency on removed packages

### Approach 5: Upgrade vulnerable Python packages

**Done** — upgraded on 2026-03-03:

- `pypdf` 6.6.2 → 6.7.5 (resolves 6 CVEs)
- `cryptography` 46.0.4 → 46.0.5 (resolves CVE-2026-26007)

**Effort:** Low
**Impact:** Resolved ~7 Python package CVEs
**Risk:** Low — all backend tests pass

### Approach 6: Accept and document (`.trivyignore`)

For CVEs with no fix and negligible actual risk, create a `.trivyignore` file:

```
# .trivyignore — Accepted CVEs with no upstream fix
# Reviewed: 2026-03-03
# Next review: 2026-06-03

# tar: setuid warning — no tar extraction in runtime
CVE-2005-2541

# glibc low-severity — mitigated by non-root + seccomp
CVE-2010-4756
CVE-2018-20796
CVE-2019-1010022
...
```

**Effort:** Low
**Impact:** Reduces noise in scan reports
**Risk:** None — documents acceptance, does not change security posture
