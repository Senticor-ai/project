# Epic: Offline Resilience & Airplane Mode

**Status:** In Progress (Foundation shipped, persistence pending)
**Priority:** Current
**Last updated:** 2026-03-06
**Depends on:** TanStack Query persistence, IDB-backed auth cache, Service Worker (vite-plugin-pwa)
**Related:** [Architecture](?path=/docs/engineering-architecture--docs), [Data Retention](?path=/docs/engineering-data-retention--docs)

---

## Context

Senticor targets Bundesbeamte (federal clerks) who frequently work in environments with
limited or no connectivity — courtrooms, secure facilities, public transit, and airplane
mode during business travel.

Two critical failures were identified during airplane mode testing:

1. **Capture fails offline.** The inbox capture (`useCaptureInbox`) performs an optimistic UI
   update, but when `POST /items` throws `ApiError(status: 0)`, the `onError` handler rolls
   back the optimistic record and a "Server is not reachable" error toast appears.

2. **Logout on app restart.** The `AuthProvider` calls `GET /auth/me` on mount. When offline,
   this fails silently, setting `user: null` and redirecting the user to the login page —
   even though the HTTP-only JWT cookie is still valid and cached data is available in IDB.

### What was shipped (Foundation — 2026-03-06)

Three fixes landed as the offline foundation:

| Fix | What | Files |
|-----|------|-------|
| **Auth resilience** | Persist last-known user to IDB; fall back when offline + `/auth/me` fails | `auth-cache.ts`, `auth-context.tsx`, `offline-storage.ts` |
| **Toast suppression** | Suppress "Server is not reachable" toasts when offline (OfflineBanner already covers it) | `ToastProvider.tsx` |
| **Mutation resilience** | Prevent optimistic rollback on offline captures; add `mutationKey` to all 6 create mutations | `use-mutations.ts` |

**Test coverage:** 19 new unit/integration tests across 4 test files (1681 total).

### What remains (this epic)

The foundation keeps the app usable during a single session, but items captured offline are
lost if the user closes the tab before reconnecting. Full offline support requires persistent
mutation queues, Background Sync, and UI indicators for pending items.

## Goals

1. **Persistent offline mutation queue** — captures survive tab close and replay on reconnect.
2. **Pending item indicator** — show a sync icon on items that haven't reached the server yet.
3. **Background Sync** — use the Background Sync API (where available) to sync pending mutations
   even after the tab is closed.
4. **Reconnection sync** — automatically invalidate stale queries and replay queued mutations
   when the browser comes back online.
5. **Conflict resolution** — handle edge cases where offline mutations conflict with server state.

## MVP Scope

### 1) Persistent mutation queue via IDB

Serialize pending (failed-offline) mutations to IndexedDB so they survive tab close.

**Approach:** TanStack Query v5 supports mutation dehydration. Configure the
`PersistQueryClientProvider` to also persist mutations that have `mutationKey` set.
On app restart, rehydrated mutations resume when online.

**Key constraint:** Mutation variables must be JSON-serializable. All create mutations
already pass JSON-LD objects — this is satisfied.

**Implementation:**
- Extend `createIdbPersister()` in `offline-storage.ts` to include mutation state
- Add `dehydrateMutations: true` to `PersistQueryClientProvider` options
- Add `resumePausedMutations` call on `online` event in `AppBootstrap.tsx`
- Configure `shouldDehydrateMutation` to only persist mutations with a `mutationKey`

### 2) Pending item indicator

Optimistic items already have `item_id` starting with `"temp-"`. Use this as a visual marker.

**Implementation:**
- Add `isOptimisticItem(record: ItemRecord): boolean` helper to `model/types.ts`
- In `ActionRow.tsx`, show a subtle cloud-upload icon (`cloud_upload`) when `isOptimisticItem`
- Use `text-muted` color and pulse animation for the indicator
- Hide indicator when item gets real `item_id` from server (on mutation success)

### 3) Background Sync integration

The `sync-manager.ts` already registers Background Sync tags. Wire it up to mutation failures.

**Implementation:**
- In `use-mutations.ts` `onError`, when offline: register a Background Sync tag
- In the service worker (`sw.ts`), handle the `sync` event by reading pending mutations
  from IDB and replaying them
- Graceful degradation: Background Sync is only available on Chromium-based browsers;
  on Safari/Firefox, rely on the in-tab `online` event replay

### 4) Reconnection sync

When the browser comes back online, automatically sync state.

**Implementation:**
- Add `useOfflineSync()` hook that listens to `window.addEventListener("online", ...)`
- On reconnect:
  1. Resume paused mutations via `queryClient.resumePausedMutations()`
  2. Invalidate `ITEMS_QUERY_KEY` to refresh from server
  3. Retry failed-offline mutations from the persistent queue
- Show brief "Synchronisierung..." toast during replay

### 5) Conflict resolution (lightweight)

Offline-created items have pre-generated `@id` (canonical_id). The server uses these
as idempotency keys. If a mutation is replayed after the server already processed it
(e.g., via Background Sync), the server should return the existing record.

**Implementation:**
- Backend: Make `POST /items` idempotent on `@id` — if a record with the same `@id`
  exists, return it instead of creating a duplicate
- Frontend: `onSuccess` already replaces optimistic records by `canonical_id`, so
  duplicate responses merge cleanly

## Product Behavior

### Offline capture flow

1. User types text in "Capture a thought..." input
2. `useCaptureInbox` fires → optimistic record with `temp-xxx` id appears in inbox
3. `POST /items` fails with network error
4. `onError` detects offline → keeps optimistic record (no rollback)
5. Toast suppressed (OfflineBanner visible)
6. Pending mutation serialized to IDB
7. Item shows cloud-upload indicator
8. User can continue capturing more items — all queue in IDB

### Reconnection flow

1. Device comes back online → `online` event fires
2. `useOfflineSync` resumes paused mutations
3. Pending mutations replay sequentially (server creates items)
4. `onSuccess` replaces `temp-xxx` records with real server records
5. Cloud-upload indicators disappear
6. Brief "Synchronisiert" success toast
7. Query invalidation refreshes full item list

### Auth during prolonged offline

1. App starts offline → `AuthApi.me()` fails → cached user from IDB used
2. User works normally with cached data
3. On reconnect → `AuthApi.me()` succeeds → cached user updated
4. If JWT expired during offline period → refresh token exchange occurs automatically
5. If refresh token also expired → user prompted to re-login (clear message, no silent fail)

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| IDB storage quota exceeded on mobile | Limit pending mutation queue to 100 items; show warning at 80 |
| Mutation replay order matters (e.g., create then triage) | Replay mutations in FIFO order; each waits for previous to complete |
| Background Sync not available on Safari/Firefox | Graceful degradation to in-tab `online` event sync; document limitation |
| Stale auth cache used for too long offline | Set max age on cached user (24h); clear on explicit logout |
| Conflict between offline capture and server-side changes | Idempotent `@id` prevents duplicates; last-write-wins for updates via ETag |
| Large IDB mutations slow app startup | Lazy-load mutation queue; don't block render on rehydration |
